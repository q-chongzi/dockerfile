系统架构

架构说明

 

数据源

关系型数据库

讯枢支持mysql、sql server、oracle、db2、达梦、神通、金仓、greenplum、postgresql、informix等关系型数据库，对于关系型数据库，讯枢纽以jdbc的方式建立连接，然后用数据库元数据实现对数据的一致操作。

外部数据

外部数据主要指没有统一规范的、某公司特有的格式数据，包括：csv、xls、xml、ws(web services)、hive、spark等等，这些数据需要根据业务情况开发对应的外部数据类，外部数据类封装了这些数据的所有操作，讯枢使用外部数据类统一使用这些数据。

讯枢RESTFul

在讯枢中配置的功能以RESTFul的方式提供给第三方，讯枢RESTFul也作为讯枢的一种数据源，可以完成类似数据整合再整合、层层上报等业务场景。

 

数据层

枢纽数据层

讯枢用jdbc元数据用于识别、探查、构建关系型数据库的操作模型，系统在启动的时候动态检查所有配置的关系型数据库，将每个数据库的表、字段以及字段的类型、长度都读到操作模型中，当需要操作某个数据库的某个表的时候，枢纽数据层根据操作模型自动生成相应的sql，操作结果以xml的形式追加到上下文中。

 

此外，对于关系型数据库中的复杂数据可以开发功能数据类来处理并加载到上下文。

 

反射加载

在需要操作外部数据时需要在功能配置的时候选择外部数据类，选择的外部数据类在运行过程中会以反射加载的方式被调用，操作结果以xml的形式追加到上下文中。

 

HTTP Client

讯枢RESTFul通过HTTP协议访问，讯枢内建了RESTFul的HTTP Client，可方便地使用其他讯枢提供的数据服务。

 

逻辑层

功能配置

功能配置包括：功能目录、目录响应、目录数据、数据参数、数据取值、数据引用、执行条件、查询条件等，具体配置参见后续的内容。

 

上下文

所有的功能配置都是为了将通过枢纽数据层、反射加载、HTTP Client获取到的数据以xml的格式组织起来形成中间数据。

上下文可通过xsl语言转换成前端开发需要的html、xml、json、纯文本、svg、pdf等，前端将接收到的数据用符合html标准的方式展现出来。

上下文也可以通过xsl语言转换成外部系统需要的接口协议数据、转换成查询数据库操作的条件或更改数据库的数据，以及转换成调用功能数据类、外部数据类、HTTP Client访问讯枢等的方法的参数。

 

转换引擎

转换引擎根据功能配置加载xsl样式单对上下文进行转换，或者根据功能配置加载系统样式单生成xsl样式单，然后再加载xsl样式单对上下文进行转换。

 

操作日志

操作日志详细记录了每次操作的过程和结果，便于排查系统系统，记录操作日志以多线程后台运行的方式执行，对业务执行几乎没有影响。

 

服务层

讯枢RESTFul

在讯枢中配置的所有功能都以RESTFul的方式提供外部访问，并通过帐号权限设置对需要授权的功能进行控制。

资源配置

系统字段

在一些表中一些有共同作用的字段，特在说明整个功能配置之前预先做解释，这些都是系统字段，不需要在新增时对这些字段进行配置，仅在查询、修改时可能需要用它们作为查询条件的一部分。

 

code：在讯枢运行中特殊的主键，使用它作为表的主键能简化配置，具体如何简化配置的参见后面对目录数据配置的内容。

is_active：记录的逻辑删除标记字段，当is_active为false时表示本条记录已被逻辑删除，表中可以没有is_active字段，这个时候的停用/删除操作执行的是物理删除。

change_from：当code、is_active、change_from三个字段同时出现在某个表并依次是前三个字段，讯枢会认为这个表有历史表，在执行修改操作时，会将当前操作的记录尝试新增到相同表名加上“_h”的历史表中，然后将新增到历史表中记录的code作为change_from的值。

change_index：用于记录数据第几次改变。

change_number：用于记录数据改变的次数。

secret：用于根据业务逻辑设置数据的保密权限，保密权限在组织机构中才会生效，保密权限见下一节内容。

secrecy：在需要的时候，可以让记录创建者设置数据的保密权限。

builder：记录的创建者，在需要登录使用的应用中创建者就是登录者的帐号。

owner：记录的所有者，不一定是记录的创建者。

start_date：记录生成时的时间戳。

end_date：记录关闭时的时间戳，记录关闭时也就是被移动到历史表的时候，所以正式数据中这个字段会一直为空。

connection_code：记录所在数据库的数据库连接的code，由于讯枢内建分库分表，connection_code将简化配置，具体如何简化配置的参见后面对目录数据配置的内容。

资源配置

功能配置主要目的是行程外部能访问的RESTFul服务，由于讯枢RESTFul和数据库都是讯枢的数据源，再加上配置入口相同，所以数据库的配置和讯枢的配置被放在一起，外部数据、功能数据通过预先在基本设置中的驱动处理中添加对应的类，然后再在配置讯枢的目录数据时根据业务需要选择使用，数据库元数据在配置了数据库以后就成为讯枢的内建数据了，也是在配置讯枢的目录数据时根据业务需要选择使用。

 

在讯枢配置过程中会使用数据库资源，因此应先配置数据库。

数据库定义

讯枢内建关系型数据库分库分表的支持，提供分布式查询支持，该功能对使用者是透明的，使用者仅在配置的时候选择表名就可以将分布在n个数据库中的数据加载到上下文中，数据库定义数据对应存储在hinge数据库的resource_decomposing[资源定义]表中。

 

数据库定义是对分布到多个数据库的统一定义，类似是数据库连接的模板，使用者通过数据库定义配置功能中需要使用的数据，讯枢通过关联到数据库定义的数据库连接操作数据。

 

在资源类型中选择“数据库”后打开数据库资源定义界面：

在右侧资源定义的新增区域录入数据库的各个连接参数，其中：资源名字就是给数据库起一个名字，方便其他人理解；资源标识是数据库的唯一标识，数据库连接和数据库定义要使用相同的资源标识；资源类型显示用，不可修改；驱动处理选择对应数据库的驱动程序名字，缺少的数据库驱动可以在基本设置中的驱动处理中添加；服务地址是数据库服务器的IP；服务端口是数据库提供服务的端口（mysql通常为3306、oracle通常为1521）；访问帐号是为使用数据库建立的帐号；访问密码是访问帐号的密码；连接模板是数据库建立jdbc连接的模板（mysql可以用jdbc:mysql://HOST:PORT/INSTANCE、

oracle可以用jdbc:oracle:thin:@HOST:PORT:INSTANCE，注意：模板中的HOST将被服务地址替换，PORT将被服务端口替换，INSTANCE将被数据实例替换）；方案标识通常指schema或者用户名；数据实例通常指数据库名字或者服务名；最大连接是指让讯枢为数据库建立多少个连接的数据库连接池；超时设置是指数据库连接的超时断开时间；维护电话和维护邮箱是便于讯枢在发现异常时通知相关人员解决问题，多个电话或者邮箱以逗号分隔。

在右侧功能目录的新增区域录入功能目录的各个参数，其中：目录标识用于和应用标识一起指示访问的是哪一个功能，用于指示访问功能的目录标识参数[catalog.cludove]，命名规则是操作对象.操作名字.响应方式，目录标识使用英文小写字母，操作对象通常是表名且可以多级（每级之间也用逗号分隔），操作名字可以用search/查询、build/新增|保存、modify/修改、remove|delete/停用|删除、migrate/迁移、synchronize/同步，响应方式可以用view|前端展示、process|前端处理等，命名规则一方面是要保证目录标识的唯一性，另一方面是要让人容易理解功能目的；目录名字就是给功能起一个名字，方便其他人理解；目录类型包括服务功能、数据迁移、数据同步三种，服务功能主要用于开发业务功能，数据迁移和数据同步是数据交换功能；响应类型根据前端和外部业务需要确定的数据格式来选择，需要html、xml数据时选择text/html，需要json、文本内容时选择text/plain，不需要返回时选择text/none；判断权限用于指示讯枢在处理请求时是否判断登陆者的权限，同一应用下有的功能需要判断权限，有的不需要判断权限；入口目录指示讯枢在浏览器打开服务地址时默认执行的功能，可配合入口应用一起发挥作用，也可以单独发挥作用；是否暂停用于在运行时停止或启用某个功能；循环取值用于定义配置功能目录的样式单是否循环输出目录响应的配置内容，在手写功能目录的样式单时不要定义循环取值；取值条件是为了配合循环取值的定义，合在一起可以让讯枢输出<xsl:for-each select="[取值条件]">...</xsl:for-each>标记；功能描述就是给功能目录的功能进行简单说明，方便其他人理解；定时运行用于控制该功能是否定时执行，定时功能一般用于后台程序、迁移、同步等；定时年份用于规定定时执行的年份，不填内容则默认为每年，需要规定多个年份时用逗号分隔；定时月份用于规定定时执行的月份，不填内容则默认为每月，需要规定多个月份时用逗号分隔；定时日期用于规定定时执行的日期，不填内容则默认为每天，需要规定多个日期时用逗号分隔；定时时钟用于规定定时执行的时钟，不填内容则默认为每时，需要规定多个时钟时用逗号分隔；定时分钟用于规定定时执行的分钟，需要规定多个分钟时用逗号分隔，需要每分钟执行时需要特别输入NA。

在应用设置界面点击左侧的数据名字，就会在当前位置根据操作名字展开并显示该目录数据的数据参数、查询条件、数据排序、数据取值、数据引用、执行条件：配置数据参数相当于手写“目录标识.处理顺序.parameter.xsl”文件，用于为目录数据的操作传递一组数据，用xml格式输出，用parameters做根标签，用parameter输出组标签，然后在parameter下输出参数标签及对应的值，值可以从上下文中选取，也可以是固定文本，在数据参数的样式单中仅能输出一组parameter标签；配置查询条件相当于手写“目录标识.处理顺序.expression.xsl”文件，用于给查询、迁移、同步、修改操作配置一个sql条件；配置数据排序相当于手写“目录标识.处理顺序.order.xsl”文件，用于给查询、迁移、同步操作配置一个order子句；配置数据取值相当于手写“目录标识.处理顺序.value.xsl”文件，用于为保存、修改操作传递n组参数，以便在数据库中插入、修改（保存操作时接收到code参数视为修改操作，否则视为插入操作）时输出修改内容字段的值，输出标签和数据参数一样，参数标签保持大写且与表的字段一致；数据引用不需要手写或配置样式单来转换数据，它用于在保存、修改操作时无法用手写或配置数据取值的样式单生成字段值时为某字段指定数据加工类，数据加工类是实现com.cludove.dirigible.hold.QuoteProcessor接口的java程序，指定数据加工类以后字段的值就由该java程序生成，上下文中的参数部分会自动以“Map<String, Object> parameters”参数传递给功能数据类的transform方法，也可以用数据参数样式单生成为其传递明确的“Map<String, Object> parameters”参数，更详细的说明请参见专门的章节；配置执行条件相当于手写“目录标识.处理顺序.condition.xsl”文件，用于为修改操作传递n组（与数据取值数量相等）参数，以便在数据库中修改（保存操作时接收到code参数视为修改操作）时输出修改条件字段的值，输出标签和数据参数一样，参数标签保持大写且与表的字段一致。

 

在右侧目录数据的新增区域录入目录数据的各个参数，其中：数据名字就是给功能起一个名字，方便其他人理解，以及用于在功能目录导航中显示，数据名字也被用于权限分组，同组权限名字一致归集；根据业务需要选择数据类型。

 

当数据类型选择数据库时，还需要录入的参数为：资源定义就是选择一个在之前定义的数据库定义；数据库表在选择资源定义后可以选择需要操作的表名；操作名字根据业务需要选择查询、保存、修改、停用、迁移、同步其中一个；是否分页只对查询、迁移、同步有效，由于控制每次返回的数据数量；判断权限用于指示讯枢在处理请求时是否判断登陆者的权限，同一功能目录下有的目录数据操作需要判断权限，有的不需要判断权限；处理顺序用于控制目录数据的操作顺序，当后续数据处理需要依赖其他数据时必须保证执行顺序；数据行数用于配合是否分页控制每次操作返回的数据数量（分页也可以通过前端传递pageNumber.cludove/页号、recordNumber.cludove/每页记录数来控制，pageNumber.cludove/页号、recordNumber.cludove/每页记录数优先于是否分页和数据行数配置，两方面设置都缺失的情况下默认显示第一页的15条数据，相当于pageNumber.cludove=0、recordNumber.cludove=15）；唯一描述多数情况下是指主键字段，保持大写；循环取值用于定义配置目录数据的样式单是否循环输出数据取值、执行条件的配置内容，在手写目录数据的样式单时不要定义循环取值；选取条件是为了配合循环取值的定义，合在一起可以让讯枢输出<xsl:for-each select="[选取条件]">...</xsl:for-each>标记，用循环取值和选取条件以及数据取值、执行条件的相关配置可以代替数据取值、执行条件的样式单，在手写目录数据的样式单时不要定义选取条件。

目录响应

配置目录响应的目的是让讯枢根据配置输出“目录标识xsl”内容（不会保存成实体文件），然后，就不再需要手写“目录标识.xsl”文件，配置目录响应优先级低于手写“目录标识.xsl”文件，目录响应对应存储在hinge数据库的catalog_response[目录响应]表中。

 

除了可以手写功能目录样式单之外，还可以配置目录响应来生成功能目录样式单（目录响应配置生成的功能目录样式单可以输出两层标记的内容，需要输出多层标记的内容需要手写功能目录样式单），配置目录响应的作用就是让讯枢根据配置生成原来需要手写的功能目录样式单，目录响应生成的功能目录样式单依然会include一个common.xsl文件，所以common.xsl需要做相同的处理。

 

目录响应可以根据不同业务条件输出不同的内容（格式必须保持一致，否则应另建新的功能目录），当需要输出不同内容时就建立一个响应顺序，当建立了多个响应顺序后，讯枢会用<xsl:choose>根据输出条件输出对应相应方案的内容，只有一个响应顺序时不会输出<xsl:choose>以及<xsl:when test="[输出条件]">：

在建立响应顺序时需要录入响应顺序的各个参数，其中：处理顺序用于在有多个响应顺序时确定<xsl:when test="[输出条件]">的输出位置，目录响应可根据需要（输出条件）返回不同的内容；输出条件用于为当前处理顺序定义一个输出条件，当只有一个处理顺序时，输出条件不会生效直接根据当前配置输出查询条件，当有多个处理顺序时，首先讯枢根据配置输出<xsl:choose>...</xsl:choose>标记，然后在xsl:choose下用<xsl:when test="[输出条件]">...</xsl:when>标记根据当前配置输出响应顺序；循环取值用于定义当前响应顺序中是否循环输出响应内容；选取条件是为了配合循环取值的定义，合在一起可以让讯枢输出<xsl:for-each select="[选取条件]">...</xsl:for-each>标记，用循环取值和选取条件以及响应顺序的相关配置可以代替功能目录的样式单；响应格式包括json、xml、html、text，根据业务需要选择；响应格式为json、xml、html时需要将最外层标记录入到起始符号和结束符号中；起始符号是用于包裹输出内容的特定符号，响应格式为json时可能是“{”、“[”，响应格式为xml、html时是“<”，响应格式为text时可根据需要自定义；结束符号是用于包裹输出内容的特定符号，响应格式为json时可能是“}、“]”，响应格式为xml、html时是“>”，响应格式为text时可根据需要自定义。
